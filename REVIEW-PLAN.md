# Review Findings & Fix Plan

Audit of the merged repo performed on 2026-03-11. Each finding is prioritized and includes the affected files, line references, and a concrete fix description.

---

## Critical

### 1. Invalid JSON in opencode.json

**Files:** `opencode.json:166`
**Problem:** Trailing comma after the last command block makes the file invalid JSON. The core install path (`README.md:18`, `docs/opencode-setup.md:3`) tells users to copy this file verbatim — it will fail on parse.
**Fix:** Remove the trailing comma at `opencode.json:165` (after the `mem-clear` command block closing brace).

## High

### 2. repo-pack / repo-pack-slim reference unsupported outputPath parameter

**Files:** `opencode.json:42-50`, `docs/opencode-commands.md:15-16,26`
**Problem:** The `repo-pack` and `repo-pack-slim` command templates tell the agent to call `pack_codebase` with `outputPath: ./repomix-output.xml`. The current repomix MCP tool surface does not expose an `outputPath` parameter — it only supports `directory`, `compress`, `includePatterns`, `ignorePatterns`, `topFilesLength`, and `style`. In practice, output goes to a temp directory (`/tmp/repomix/...`) and must be manually copied. The docs claim the commands write directly to `./repomix-output.xml`, which is misleading.
**Fix options:**
- **Option A:** Remove `outputPath` from the command templates. Update the template prompt to instruct the agent to copy the output file into the project directory after packing (matching what the skill files already do).
- **Option B:** If a future repomix version adds `outputPath` support, keep the templates but gate them behind a version note.
- **Recommendation:** Option A. Update `opencode.json` templates and `docs/opencode-commands.md` descriptions to reflect the actual two-step flow (pack → copy).

### 3. Spec-Kit install guide missing Node.js prerequisite

**Files:** `docs/speckit-install.md:7-11`, `README.md:18-23`
**Problem:** The quick-start tells all users to copy the shared `opencode.json`. The Spec-Kit install guide only lists Python + uv as requirements. But the shared config uses three `npx`-backed MCP servers (`repomix` at `opencode.json:13`, `memory` at `opencode.json:21`, `sequential-thinking` at `opencode.json:25`), which require Node.js. A Spec-Kit user following only their install guide will have broken MCP servers.
**Fix:** Add a "Shared prerequisites" note at the top of `docs/speckit-install.md` (and optionally `docs/openspec-install.md` for symmetry) stating that Node.js is required for the shared OpenCode config, or add a prerequisites section to the root `README.md` quick-start.

### 4. Skill location ambiguity (.claude/skills vs ~/.config/opencode/skills)

**Files:** `docs/openspec-install.md:90`, `docs/speckit-install.md:70`, `docs/opencode-setup.md:300-322`
**Problem:** The shared OpenCode docs say skills live under `~/.config/opencode/skills/` (global) and `.opencode/skills/` (project). But the OpenSpec and Spec-Kit install guides say those tools inject slash commands as skills into `.claude/skills/`. These are different mechanisms that coexist, but the docs never explain the distinction. A reader may think one contradicts the other.
**Fix:** Add a brief clarification paragraph in `docs/opencode-setup.md` (config file locations section) explaining:
- `.claude/skills/` — slash-command skills generated by spec tools (OpenSpec, Spec-Kit). These are tool-specific and managed by the tool's `init` command.
- `~/.config/opencode/skills/` — global OpenCode skills (like `spec-review`). These are manually installed (symlinked from this repo).
- `.opencode/skills/` — per-project OpenCode skills (committed to Git).

### 5. Agent templates framed as global rules but installed as project files

**Files:** `agents/openspec.md:1-3`, `agents/speckit.md:1-3`, `README.md:19-21`, `docs/opencode-setup.md:234-255`
**Problem:** `README.md` tells users to copy agent templates into a project root as `AGENTS.md`. Both templates begin with "# Global OpenCode Rules" and "Personal defaults that apply to every project and session." But the setup docs explain that global rules go in `~/.config/opencode/AGENTS.md` (personal, all projects) while project `AGENTS.md` is team-shared (committed to Git). The current wording makes the templates sound like global config when they are meant to be project files.
**Fix options:**
- **Option A (recommended):** Reword both templates. Change the title to "# Project Agent Rules" and the subtitle to something like "Spec-driven workflow defaults for this project." Remove "Personal defaults that apply to every project and session."
- **Option B:** Keep the templates as global rules and change the README to instruct copying to `~/.config/opencode/AGENTS.md` instead of the project root. But this means the spec-specific workflow rules won't be shared with the team via Git.
- **Recommendation:** Option A. These are project-level files.

## Medium

### 6. Missing mkdir -p in install steps

**Files:** `README.md:18,33-36`, `docs/opencode-setup.md:316-321`
**Problem:** Install instructions use `cp` and `ln -sfn` targeting `~/.config/opencode/` and `~/.config/opencode/skills/`, but never create those directories. On a clean machine (or a machine where OpenCode hasn't been run yet), these commands will fail.
**Fix:** Add a prerequisite step before the first `cp`/`ln` command:
```bash
mkdir -p ~/.config/opencode/skills
```

### 7. Workflow doc entrypoint missing shared command reference

**Files:** `docs/opencode-workflows.md:5`
**Problem:** The intro line says "For the full command references, see openspec-commands.md or speckit-commands.md." But the workflows heavily use shared OpenCode commands (`/repo-overview`, `/repo-stats`, `/repo-auth`, `/ast-find`, `/c7-how`, `/gh-docs`, `/mem-save`, etc.) that are documented in `docs/opencode-commands.md`, not in the tool-specific references.
**Fix:** Add `opencode-commands.md` to the intro line:
```
For the full command references, see [opencode-commands.md](./opencode-commands.md) (shared commands),
[openspec-commands.md](./openspec-commands.md) (OpenSpec), or [speckit-commands.md](./speckit-commands.md) (Spec-Kit).
```

### 8. Undocumented plugin dependency

**Files:** `opencode.json:3`
**Problem:** The config enables plugin `@tarquinen/opencode-dcp@latest`, but the repo never explains what it is, why it is needed, or how to install it. Users copying the config will silently depend on an unexplained plugin.
**Fix options:**
- **Option A:** Add a brief note in `docs/opencode-setup.md` (near the MCP servers section or in a new "Plugins" section) explaining what the plugin does.
- **Option B:** If the plugin is optional/personal, remove it from the shared template and document it as an optional addition.
- **Decision needed:** Determine whether this plugin should be part of the shared config or is a personal preference.

### 9. Context7 tool naming drift (get-library-docs vs query-docs)

**Files:** `opencode.json:121-147`, `agents/openspec.md:23`, `agents/speckit.md:23`
**Problem:** The command templates and agent rules reference `get-library-docs`, but the current MCP tool surface exposes `query-docs`. This may be an OpenCode command-layer alias, but it is unverified.
**Fix:** Validate which name the Context7 MCP actually exposes in the current OpenCode version. If it is `query-docs`, update the references in `opencode.json` and both agent templates. If OpenCode aliases it, add a note.

### 10. Spec-Kit new-feature flow inconsistency

**Files:** `docs/opencode-setup.md:147-156`, `docs/opencode-workflows.md:82-121`
**Problem:** The setup doc's recommended Spec-Kit new-feature flow includes `/speckit.constitution` as step 2. The fast-path workflow in the workflows doc starts directly at `/speckit.specify`, omitting `/speckit.constitution`. This may confuse first-time users who don't yet have a constitution.
**Fix:** Add a note to the fast-path workflow (Workflow 2, Spec-Kit section) stating that `/speckit.constitution` should be run once per project before the first `/speckit.specify`. Something like:
```
> If this is the first time using Spec-Kit in this project, run `/speckit.constitution` first.
```

### 11. README skills link points to OpenSpec-specific anchor

**Files:** `README.md:46`
**Problem:** The "See opencode-commands.md § Skills" link points to `#spec-review--openspec-verification-skill`, which is the OpenSpec-specific skill section. Users looking for the Spec-Kit skill would land on the wrong section.
**Fix:** Either link to a neutral anchor that covers both skills, or list both links explicitly.
